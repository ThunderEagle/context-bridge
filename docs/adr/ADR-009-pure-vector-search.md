# ADR-009: Pure Vector Search over Hybrid (BM25 + Vector)

**Date:** 2026-06-21  
**Status:** Accepted  
**Deciders:** Scott Williams

---

## Context

The `memory_search` MCP tool returns the most relevant memories for a given query. Two search strategies are viable with SQLite:

**Pure vector search** — embed the query with all-MiniLM-L6-v2, find the k nearest neighbours in `memories_vec` via cosine similarity, return top-k results ranked by distance.

**Hybrid search** — combine vector similarity with BM25 keyword relevance (SQLite FTS5 extension) and merge the ranked lists via Reciprocal Rank Fusion (RRF). Used widely in RAG pipelines to compensate for weak embedding models on long or sparse documents.

## Decision

We will use **pure vector search** via sqlite-vec for all `memory_search` queries in v1. FTS5 and RRF are not implemented.

The two conditions that motivate hybrid search are absent in this use case:

1. **Short, semantically dense content.** Memories are short snippets (a few sentences at most) written in natural language. `all-MiniLM-L6-v2` performs well on this content type — it was trained on sentence pairs and is specifically designed for semantic similarity on short text.

2. **Machine-generated queries.** Search queries are generated by Claude, not typed by a developer. Claude produces semantically precise queries rather than keyword fragments. The keyword fallback that BM25 provides (matching exact tokens when the embedding space disagrees) has no advantage when the query is already semantically formulated.

Hybrid search would add:
- FTS5 virtual table and tokenization configuration
- RRF merge logic (ranking combination math)
- Two result sets to fetch and sort instead of one
- Additional pipeline latency on every search call

None of that complexity solves a problem that exists in this use case.

## Consequences

### Positive
- `memory_search` is a single sqlite-vec KNN query — minimal latency, straightforward implementation
- No FTS5 tokenization configuration, stop word lists, or ranking weight tuning
- Storage schema is simpler: no `memories_fts` virtual table, no `content` mirror for FTS
- The right call to revisit: if semantic search recall proves inadequate in practice (measurable, not hypothetical), hybrid search can be layered on without changing the public interface

### Negative
- Exact-token queries like `"ADR-007"` may not surface relevant memories if the embedding model does not preserve the token precisely in the vector space — acceptable for this use case where such queries are rare and Claude will naturally reformulate them semantically

### Neutral / Trade-offs
- Result ranking is purely by cosine distance — no relevance feedback, no re-ranking by recency or tag, no LLM-based re-ranker. Sufficient for v1; re-ranking is a v2 concern if recall quality data warrants it.

## Alternatives Considered

| Option | Reason Rejected |
|---|---|
| Hybrid BM25 + vector with RRF | Adds implementation complexity (FTS5 + RRF merge) without addressing a demonstrated recall problem in this specific content/query profile |
| BM25 only (FTS5) | No semantic understanding; fails on paraphrase and synonym matching — the core value proposition of the service |
| LLM re-ranker (call Claude inside the MCP tool) | High latency and cost per query; re-ranking presupposes a recall quality problem that doesn't exist yet |
| In-process cosine scan (no sqlite-vec) | O(n) per query on the full vector set; correct for small datasets but doesn't scale and misses future ANN index support from sqlite-vec |
